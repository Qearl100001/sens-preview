# Layout / Grid Foundation

> 主要来源：Figma `Sens.Design 栅格布局 v0.3 220505`、`/Users/liyuwen/Desktop/DESIGN.md`、`src/features/tiktok-ads-connections/DataSourceManagementPage.tsx`、`src/ui/SensPageTitleBar.tsx`、`src/design-system/components/base/drawer.design.md`。
> 当前状态：第一版规则沉淀；先定义页面级布局、栅格列数、断点行为和局部容器边界，不代表所有页面都已完成统一接入。

## 1. 定位

Layout / Grid Foundation 负责统一页面级布局、内容容器、栅格列数、断点行为和局部区域排布规则。

它回答的不是“某个组件长什么样”，而是：

- 页面内容应该落在哪种骨架里。
- 内容区如何分栏。
- 不同宽度下页面如何响应。
- Drawer / Modal / 管理页 / 列表页分别用哪种容器结构。

后续页面和业务样板间不应该各自发明新的页面骨架，而应该优先从这一层选型。

## 2. 基础原则

- 栅格最小单元使用 `4px`。
- 页面级布局使用 `20 栏栅格`。
- Drawer / Modal 内部布局使用 `12 栏栅格`。
- 默认面向桌面端，当前主验收范围为 `1280-1920`。
- 先保证信息层级稳定，再追求自适应弹性。
- 页面骨架优先复用，不允许业务页面随意发明新的整页布局。

## 3. 断点与布局行为

当前记录的常用桌面宽度：

| 断点 | 宽度 | 布局策略 |
|---|---:|---|
| XS | 1280 | 静态布局，内容保持固定结构，必要时出现横向滚动 |
| SM | 1366 | 响应式布局，进入桌面端常规验收区间 |
| MD | 1440 | 响应式布局，设计画布默认宽度 |
| LG | 1600 | 响应式布局，大桌面内容更舒展 |
| XL | 1920 | 大屏优化 |

适配策略：

- 低于 `1280`：静态布局，保持结构稳定，不强行压缩页面。
- 高于或等于 `1280`：响应式布局，主内容区按 `20 栏` 规则组织。
- 重要组件区域：允许做自适应布局，例如表格、卡片区、左右双栏信息区。
- 抽屉：当前以组件级宽度承接，常用 `small = 432px`、`medium = 864px`；内部内容仍按 `12 栏` 理解。

## 4. 页面级布局类型

### 4.1 T 型布局

适用：

- 三级落地页
- 有侧边导航或分类导航的管理页

结构：

- 左侧导航区
- 右侧主内容区

规则：

- 侧边导航展开时使用固定宽度，当前记录值为 `200px`。
- 主内容区承接标题栏、筛选区、内容区。
- 内容区内部再按 `20 栏` 或局部容器布局继续拆分。

### 4.2 上下布局

适用：

- 四级页面
- 无侧导的独立管理页
- 下钻详情页

结构：

- 顶部标题 / 操作区
- 下方内容区

规则：

- 标题区不承担复杂分栏。
- 内容区可以继续拆成单栏、双栏、列表 + 详情、表单 + 说明等局部结构。
- 当前页面标题栏默认高度记录为 `72px`，带辅助文案时使用扩展高度。

## 5. 栅格层级

### 5.1 页面主栅格

- 页面主内容使用 `20 栏`。
- 主栅格用于决定大区块宽度关系，不要求每个小组件都显式占列。
- 卡片区、筛选区、表格区、说明区都应先挂到主栅格或其派生容器上。

### 5.2 局部容器栅格

适用：

- 卡片列表
- 左右双栏信息区
- 映射配置区
- 标题区下的筛选操作组合

规则：

- 局部容器可以在主栅格内部再做二次分栏。
- 局部分栏优先服务内容结构，不追求机械等分。
- 卡片列表可以按 `4` 列或 `5` 列候选排布，但应由页面容器决定，不由 Card Foundation 决定。

### 5.3 Drawer / Modal 栅格

- Drawer / Modal 内部使用 `12 栏`。
- 适合表单、说明、辅助信息、分组内容的排布。
- 不把整页 `20 栏` 直接搬进抽屉内部。
- 当前 TikTok 创建连接抽屉优先按单栏表单理解，后续再补 `12 栏` 细分规则。

## 6. 整页布局和局部容器的边界

适合放在整页布局层决定的内容：

- 是否使用 T 型布局或上下布局
- 是否有侧边导航
- 标题区和内容区的上下关系
- 页面内容区是单栏、双栏还是卡片区 + 表格区

适合放在局部容器层决定的内容：

- 卡片每行几列
- 左右面板的比例
- 表单说明和输入区的内部分组
- 筛选区按钮和搜索框的局部排布

约束：

- 不要用 Card、Table、Form 自己反向定义整页骨架。
- 不要把局部组件间距问题误当成页面栅格问题。

## 7. 常见页面骨架建议

| 页面类型 | 建议骨架 |
|---|---|
| 管理入口页 | 上下布局 + 卡片网格 |
| 列表页 | 上下布局 + 筛选区 + 表格区 |
| 空态页 | 上下布局 + 居中内容容器 |
| 创建 / 编辑 Drawer | 抽屉容器 + `12 栏` 内部表单区 |
| 列表 + 详情 | 上下布局或左右双面板 |
| 映射 / 配置页 | 主内容区 + 局部双栏 / 分组块 |

## 9. 代码落地规则

- 当前项目还没有独立的 Layout / Grid token 生成链路。
- 页面骨架优先通过 foundation 规则和组件容器落地，不在业务页随意手写新的整页结构。
- 局部布局可使用现有 `spacing/*`、组件常量和容器样式承接，但要能解释来源。
- 抽屉宽度当前仍是组件级尺寸规则，不直接提升为 foundation token。
- 如果后续反复出现相同的页面边距、侧导宽度、内容最大宽度或局部列比，再评估是否进入 token 或统一 helper。

## 10. 与其他 Foundation 的边界

- `Spacing` 负责间距，不负责页面骨架。
- `Size` 负责固定宽高，不负责页面分栏。
- `Card` 负责卡片容器，不负责卡片在一行排几张。
- `TitleBar` 负责标题区内部结构，不负责整页骨架选择。

## 12. 第一阶段验收

第一阶段只看 3 件事：

- 页面骨架类型是否能被明确命名。
- `20 栏 / 12 栏 / 1280+` 这些规则是否能指导真实页面。
- Layout / Grid 和 `Spacing` / `Card` / `TitleBar` 的边界是否清楚。
