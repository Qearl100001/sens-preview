# 设计系统 skill · 组件优化检查单 Review Checklist

> 用途：把已有组件逐个优化到统一水准的验收基线。也可作为新组件完成后、或 Cursor 升级后重跑的验收标准。
> 用法：**一次只过一个组件**，先逐条输出"核对结论"，确认后再改；不新增功能，不擅改公共 `theme.ts`。

## 优化前先读（标准来源）
- `conventions.md`（实现约定）、`icons.md`、`theme.ts`、`i18n`，以及该组件自己的 `.md`。
- 唯一标准就是这些 skill，不凭感觉调。

## 检查单（逐条核对，输出"符合 / 不符合→怎么改"）
1. **实现方式**：是否用 theme token + antd props 驱动？有没有平行 CSS 去盖 `ant-*`？有则按 `conventions.md` 删除或收成窄作用域 + 注释。
2. **颜色**：所有颜色是否来自 token（不硬编码 hex）？语义/状态色是否用对 token（如 error/warning/link/中性色）？
3. **状态**：`disabled` / `loading` 等用 antd 真实 props，没有手动重定义；`hover` / `active` 没在真实组件里硬造。
4. **变体矩阵**：变体 × 尺寸 × 状态是否和该组件 `.md` 一致——没有拍平、没有砍列？预览态按 `conventions.md` 的"预览板"方式（从 token 取色的静态样张）。
5. **文案**：走 i18n key，没有把 Figma 图层名当文案。
6. **图标**：颜色 / 间距遵循 `icons.md`。
7. **专属规则**：该组件 `.md` 里的特有规则是否都落实（如 表格操作列=链接按钮、tabs 三类对三个不同 antd 组件、search 不用图层名当占位符、按钮"大32/小24"两种尺寸等）。

## 收口必过项（当轮有落地或决策时）

- [ ] **changelog 已补**：当日 `changelog/YYYY-MM-DD.md` 已按 `changelog/README.md` 更新；打开 `/changelog` **自动出现该日期**且正文正常。无落地且无决策可勾 N/A 并一句说明。
- [ ] **commit**：changelog 与当轮代码改动在**同一 commit**，未拆成后续补交。

**Markdown 主动更新例外（与本检查单一致）**  
用户规则默认「勿编辑用户未要求的 markdown」；**唯一例外**为 `src/design-system/changelog/` 目录——当轮满足 changelog 触发阈值时，Cursor **应主动**新建/更新当日日志，**无需用户点名**。纪律全文见 `changelog/README.md`；**其余** markdown 仍须用户明确要求或下文「改动影响范围」同步规则，照旧。

## 建议顺序（从最可能有历史遗留的开始）
按钮 → 搜索 → 表格 → 标签页 → 徽标。每过完一个，用"四维"（颜色 / 密度 / 单元格或变体类型 / 结构）+ 本检查单扫一遍，确认后再下一个。

## 两条红线
- **不要擅自改公共 `theme.ts`**：它是所有组件共享的。若某颜色不对，先怀疑"该用的 token 没用对"，而不是"token 缺了"。确实缺 token 时，停下来说明具体情况，由人判断是否新增、加哪个。
- **不要推倒重做**：检查单是用来"挑偏差并修"，不是重写。能改一行解决就别重生成，重生成易引入新偏差。

## 改动影响范围 & 同步规则（改东西前先过这一节）

**核心判断：你改的是"说法"还是"规则"？**
- 改**说法**（措辞、排版、举例、补充说明）→ 影响极小，**只动那一份文件，自己改即可，不用跟 Cursor 聊**。
- 改**规则**（尺寸、颜色语义、状态、用法约束等真实设计决定）→ 涉及的几个"真相源"都要同步，否则会出现"设计说 A、代码做 B"的不一致。

**各文件的影响范围：**
| 改了什么 | 影响范围 | 谁来改 |
|---|---|---|
| `.design.md`（设计师文档）| **不参与代码生成**，只影响 Showcase「设计规范」tab 显示 | 自己改 md 存盘，Showcase 自动更新 |
| `.md`（研发文档）| Cursor 还原/优化时会读，**直接影响生成结果** | 跟 Cursor 聊，让它落实 |
| `theme.ts`（值）| 所有组件共享，**影响面最大** | 不手改；改 `build-tokens.mjs` 重新生成 |

**真相源在哪，改动就从哪发起：**
- 设计决策的"人话版"真相源 = `.design.md` → 自己改。
- 给 AI 执行的真相源 = `.md` → 跟 Cursor 聊。
- 值（颜色/尺寸）的真相源 = `build-tokens.mjs` → 跟 Cursor 聊，它重新生成 `theme.ts`。

**一条规则变了，顺着它涉及的真相源逐个过一遍**（设计 `.design.md` / 研发 `.md` / 值 `theme.ts`），别只改一处留另两处对不上。给 Cursor 的同步指令示例：
```
[组件] 的设计规则变了：[具体改动]。请同步：
1. [组件].md 里的对应规则；
2. 涉及值就改 build-tokens.mjs 重新生成 theme.ts（不要手改 theme.ts）；
3. [组件].design.md 我自己改，你不用动。
改完告诉我动了哪些文件。
```
