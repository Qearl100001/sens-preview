# 设计系统 skill · Cursor 怎么工作 & 识破话术 How-Cursor-Works

> 给人看的"去盲盒"说明 + 给 Cursor 用的约束。配合 `conventions.md` 一起用。

## 一、样式是三层叠出来的（去盲盒）
最终呈现 = 三层从下往上叠：
1. **antd 默认**：底座。你不写任何东西时看到的就是它。
2. **你的 theme + skill**：用 token + .md 规则去"改写"第一层。**所有的"值"（颜色/间距）都该走这一层。**
3. **AI 临时手写的 CSS**：它觉得前两层没覆盖时自己写的。**几乎所有问题都来自这一层。**

目标：**所有值走第二层（token），彻底干掉第三层（手写 CSS）。**
"按 Figma 来、别被 antd 覆盖" = 把 MCP 读到的值喂进 token 让 antd 用你的值渲染，而不是让 AI 手写 CSS 去盖。

## 二、每次改样式，必须让它报告（贴进指令）
```
改完后必须报告：
1. 改了哪些文件？哪些是 theme/token，哪些是手写 CSS？
2. 有没有新增针对 ant-* 的覆盖、或手写颜色/间距值（hex / px）？有就列出并说明为什么 token 做不到。
3. 这次颜色和间距的值，分别来自哪个 token / Figma 变量？
```

## 三、安全词 vs 危险词（看它的回答判断）
**危险（它又在第三层瞎搞，立刻拦）：**
- "我加了一条 CSS / 样式去覆盖…" → 问：为什么不能用 token？
- 用了 `!important` → 基本必错，在和 antd 打架。
- "我设置了颜色 #xxxxxx / 间距 8px"（直接写值，而非"用了某 token"）→ 硬编码，改成从 token 取。
- "针对 .ant-xxx 我写了…" → 平行覆盖，违反 conventions。

**安全（走对了，放行）：**
- "通过 components.X 的某个 token 实现"
- "颜色来自 colorPrimary / 间距来自 sizeUnit"
- "删除了手写 CSS，改由主题渲染"

**一句口诀：它说"我加了/写了样式"= 危险；它说"我用了某个 token"= 安全。**
凡是它在"手动设某个颜色/间距的值"，几乎都是瞎搞；正解永远是"传对 props + 用对 token，剩下交给 antd"。

## 四、别矫枉过正（规则分两类，处理不同）
- **通用约定**（用 token 不手写 CSS、值从 token 来、不写 !important）：和组件无关，**定死、越硬越好** → 放 `conventions.md`。
- **组件专属规则**（如"二级 hover 无底""禁用=加载同色"）：是该组件的设计决定，**只写进各自的 .md，绝不提进 conventions** → 否则会拿一个组件的规则套所有组件。

避免过度矫正三原则：
1. **新组件先裸跑一遍**（按现有 theme+conventions 生成），让偏差自己暴露，再针对偏差写规则；别预写一堆"它肯定会犯的错"。
2. **只写"你和 antd 不同"的点**；antd 默认恰好对的地方就别写规则（写规则有维护成本）。
3. 遇到"这组件规则好像不一样"，先认它本就该不同 → 写进该组件 .md 即可，不用回头改别的组件。只有确认"所有组件都适用"才提进 conventions。
